## Summary

The Certification Module is an embedded domain within the existing ScholarX V2 Next.js monorepo. It automates the issuance of cryptographically signed, branded digital certificates to program participants upon satisfying configurable completion criteria (minimum watch percentage + optional quiz gate). Certificates are available for download (PDF/PNG), shareable to LinkedIn with Open Graph preview, publicly verifiable at `/verify/:id`, and optionally showcased on a SEO-indexed public portfolio page at `/u/:username/certificates`.

**Technical approach**: Pure Next.js 16 App Router domain following the established `src/domain/courses/` DDD pattern. No external queue service (DB-backed jobs table). No RSA/KMS (HMAC-SHA256 via Node.js `crypto`). PDF generation via `@react-pdf/renderer`. Asset storage on S3/R2.

**Language/Version**: TypeScript 5 / Next.js 16 (App Router)

**Testing**: Node.js built-in test runner (`node --test`), existing `pnpm test` and `pnpm test:api` scripts

**Performance Goals**: Single cert generation <5s p95; verify page LCP <1.5s on 4G; claim email delivery <60s  
**Constraints**: No Redis, no external queue; Node.js runtime declaration required on all cert API routes; `CERT_SIGNING_SECRET` env var required at runtime  
**Scale/Scope**: в‰¤200 certificates per season; ~5 admin users; ~500 monthly verifications

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| PDF generation | `@react-pdf/renderer` | No Chromium dependency; native Node.js; React component model; serverless-safe |
| Cryptographic signing | HMAC-SHA256 (Node.js `crypto`) | Sufficient for V1; zero new dependencies; fast; upgradeable to RSA in V2 |
| Async bulk jobs | DB-backed `certificate_jobs` table + polling | No Redis required; Postgres handles concurrency with `FOR UPDATE SKIP LOCKED` |
| Single cert issuance | Synchronous | в‰¤5s per cert; acceptable for event-triggered single issuance |
| QR code | `qrcode` npm package | Lightweight; server-side only; zero native deps; TypeScript-typed |
| Storage | AWS S3 / Cloudflare R2 | Existing infra direction (`.wrangler` present); pre-signed 15-min TTL URLs |
| Email | `nodemailer` (existing dep) | Already declared; no new dependency; abstract behind `IEmailService` |
| Inter-domain event | Direct service call (not HTTP) | Same Next.js process; Courses service calls `CertificateIssuanceService.issue()` directly |
| Portfolio | Opt-in, `username` + `isPublic` flag | Privacy-first; low-cost SEO growth loop |
| Verification CTA | "Learn more on ScholarX в†’" button | Converts verifiers в†’ learners; leverages existing `course.slug` |

## Implementation Phases

### Phase A вЂ” Foundation (Sprint 1, Week 1)
**Goal**: Schema in DB, contracts in code, signing working, no UI yet.

1. Install new dependencies (`@react-pdf/renderer`, `qrcode`, `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`)
2. Create `certificates-db.schema.ts` + Drizzle migration
3. Add `portfolioUsername` / `portfolioEnabled` columns to `auth.user` migration
4. Copy contracts from `specs/` в†’ `src/domain/certificates/contracts/`
5. Implement `CertificateSigningService` + unit tests (sign, verify, tamper detection)
6. Implement `CertificatesRepository` + `CompletionCriteriaRepository`

### Phase B вЂ” Issuance Core (Sprint 1, Week 2)
**Goal**: Single-cert issuance end-to-end (DB в†’ PDF в†’ S3 в†’ email).

1. Implement `CertificatePdfService` вЂ” Mentee + Mentor template components with QR embed
2. Implement `CertificateStorageService` вЂ” upload PDF/PNG, pre-signed URL generation
3. Implement `CertificateEmailService` вЂ” claim email + 7-day reminder template
4. Implement `CertificateIssuanceService` вЂ” orchestrate signing в†’ PDF в†’ storage в†’ DB в†’ email
5. Integrate with Courses domain вЂ” call `issue()` from `NextCourseEnrollmentService` on completion
6. POST `/api/certificates` вЂ” admin manual issue endpoint
7. Unit + API tests for issuance path (idempotency, criteria guard, retry on failure)

### Phase C вЂ” Verification & Claim (Sprint 2, Week 1)
**Goal**: Public-facing verification and claim flows complete.

1. `CertificateVerificationService` вЂ” HMAC verify, status check, event log
2. GET `/api/certificates/verify/[id]` route handler
3. `/verify/[id]/page.tsx` вЂ” statically generated verification page with course CTA
4. `CertificateClaimService` вЂ” atomic token consumption, wallet link
5. POST `/api/certificates/claim` route handler
6. `/certificates/claim/[token]/page.tsx` вЂ” claim page with download + share actions
7. Open Graph meta tags on verify + claim pages
8. Tests: VALID / REVOKED / INVALID paths; token expiry; double-claim

### Phase D вЂ” Admin Dashboard (Sprint 2, Week 2)
**Goal**: Full admin control over certificate lifecycle.

1. `CertificateAdminService` вЂ” list, revoke, resend, bulk trigger, CSV export
2. POST `/api/certificates/bulk` + GET `/api/certificates/jobs/[jobId]` (progress polling)
3. Admin UI: certificate list with filters, bulk issue trigger, progress bar
4. Admin UI: certificate detail page (revoke, resend actions)
5. Admin UI: CSV export button
6. Tests: bulk job creation, progress polling, revoke flow, resend flow

### Phase E вЂ” Wallet & Portfolio (Sprint 3)
**Goal**: Recipient wallet and public portfolio page live.

1. `CertificatePortfolioService` вЂ” username validation, toggle visibility, get public portfolio
2. GET/PUT `/api/certificates/portfolio/*` route handlers
3. `/my/certificates/page.tsx` вЂ” authenticated wallet with download/share actions
4. `/my/certificates/settings/page.tsx` вЂ” username setup + per-cert visibility toggles
5. `/u/[username]/certificates/page.tsx` вЂ” public SEO-indexed portfolio page
